# NeuroGlia-HD Atlas

**Cell-type-aware machine learning for Huntingtonian neuroglial state modelling from public transcriptomic atlases**

I built this repository as a research-grade bridge between biomedical AI, neurodegeneration, protein-informed feature engineering, generative representation learning, and clinically cautious model validation. The core idea is to treat neuroglial dysfunction as a measurable, multi-resolution state: single-nucleus expression at cell level, pseudobulk donor × brain-region × cell-type summaries at cohort level, and curated pathway/signature scores at biological mechanism level.

The repository is designed to run end-to-end on a MacBook M2 using CPU or Apple Metal/MPS where available, while still being structured cleanly enough to scale onto a workstation.

> **Status:** analysis-ready scaffold. Results, figures, and model artefacts are intentionally generated by the pipeline rather than committed here. I do not include fabricated metrics.

---

## Why this project exists

My previous work has focused on biomedical AI systems that learn from noisy, multimodal or biologically structured data, including generative modelling, counterfactual-style reasoning, interpretability, structural biology, medical imaging, reproducibility and healthcare access. NeuroGlia-HD Atlas extends that thread into neurodegenerative molecular modelling: instead of synthesising images or modelling protein-state shifts, this project learns interpretable disease-state representations from human and mouse neuroglial transcriptomic data.

This is not a black-box classifier with a pretty ROC curve taped to the front. It is a full analysis spine:

1. **Dataset ingestion** from public GEO accessions.
2. **Single-cell and bulk RNA-seq quality control** with audit reports.
3. **Pseudobulk construction** at donor × region × cell-type resolution.
4. **Curated neuroglial and CAG-instability feature signatures**.
5. **Nested/grouped validation** to avoid donor leakage.
6. **Classical baselines, calibrated classifiers, and neural latent models**.
7. **Cross-dataset and cross-species transfer checks**.
8. **Interpretability, ablation, perturbation-style probes, and report generation**.

## Public datasets

| Layer | Accession | Organism | What it contributes |
|---|---:|---|---|
| Human glial snRNA-seq | `GSE281069` | Human | Primary atlas for glial-state modelling across caudate nucleus, cerebellum, hippocampus and frontal cortex. |
| Mouse snRNA-seq | `GSE180294` | Mouse | R6/2 cortex and striatum model for cross-species and region-generalisation experiments. |
| Human rdRNA-seq | `GSE159940` | Human | Bulk HD prefrontal cortex samples with disease variables including CAG repeat size and age of onset. |

## Research questions

### RQ1. Can neuroglial transcriptional states separate disease and control donors without donor leakage?

Primary endpoint: held-out donor classification on pseudobulk features.

Validation: grouped cross-validation by donor, region-held-out transfer, calibration, and permutation/null-label sanity checks.

### RQ2. Which cell types carry the most transferable signal?

Primary comparisons:

- Astrocytes only
- Microglia only
- Oligodendrocyte lineage only
- All glia
- All annotated major cell types

### RQ3. Are DNA repair and CAG-instability modifier signatures detectable in glial disease-state embeddings?

Feature families include FAN1, MLH1, MSH3, PMS1, PMS2, LIG1, and broader DNA damage response modules. The intent is not to claim causal proof from expression alone, but to test whether known modifier biology appears in predictive and latent-state directions.

### RQ4. Do latent neuroglial states generalise across brain regions, species, and assays?

The project contains both classical and neural models:

- Penalised logistic regression
- Random forest
- Gradient boosting
- Calibrated ensemble
- Denoising autoencoder
- Beta-VAE-style latent model
- Optional Optuna tuning

The neural modules automatically choose `mps` when PyTorch reports that Apple Metal acceleration is available. Single-cell preprocessing remains memory-bound, so for the full human atlas I recommend closing browser tabs, using an external SSD for `data/`, and starting with the pseudobulk mode before full cell-level experiments.

## Quick start

### 1. Inspect available public data

```bash
neurogliahd registry
```

### 2. Download GEO supplementary files

```bash
neurogliahd download --config configs/default.yaml --datasets human_glia_sn mouse_r62_sn
```

### 3. Build pseudobulk features

```bash
neurogliahd pseudobulk --config configs/default.yaml
```

### 4. Train and validate baseline models

```bash
neurogliahd train-baselines --config configs/default.yaml
```

### 5. Train latent model

```bash
neurogliahd train-latent --config configs/default.yaml
```

### 6. Generate report artefacts

```bash
neurogliahd report --config configs/default.yaml
```

Or run the whole comet-tail:

```bash
bash scripts/run_full_pipeline.sh
```

## Validation philosophy

I treat every metric as suspicious until it survives a small gauntlet:

- donor-level grouped splits;
- no sample/cell leakage across folds;
- held-out brain-region experiments;
- independent dataset replication;
- null-label controls;
- permutation importance stability;
- calibration diagnostics;
- transparent failure cases.

No model in this repository is presented as a diagnostic device. The aim is to build a clean computational research platform that could support wet-lab follow-up or prospective validation after independent review.

## Big research-grade pipeline

A hardened full-run workflow is included in `BIG_PIPELINE_README.md`.

```bash
source .venv/bin/activate
caffeinate -dims bash scripts/run_big_research_grade_pipeline.sh
```

This workflow converts available downloaded public datasets, builds atlas and supervised matrices, performs nested train/tune/test evaluation, Optuna tuning, permutation controls, overfitting diagnostics, leakage/proxy screens, calibration plots and final reporting. It intentionally excludes single-class datasets from supervised HD/control classification to prevent dataset-label leakage.
